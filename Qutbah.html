01 — The Problem

English-speaking congregants miss the nuance.

When the imam quotes Quran or Hadith in Arabic, generic translation apps fall apart. They aren't tuned for formal khutbah Arabic or tajweed-style recitation, they can't tell scripture from commentary, and they either cost money on every chunk or invent the wrong citation.

⌇

Not tuned for recitation or STT mishearings — ذ/ض, ث/س

⊘

Can't reliably tell Quran from Hadith from plain commentary

⚠

Network cost on every chunk — or confidently wrong citations

The product constraint

Layer 1 (live Arabic + English) must never wait on Layer 2 (Quran / Hadith enrichment).

02 — The Solution

A two-layer pipeline, entirely on device.

One layer keeps the conversation flowing in real time. A second layer runs parallel “specialists” in the background and upgrades plain rows into canonical scripture cards the moment it's confident.

Layer 1 Instant

The live feed — never blocked

Apple on-device STT (ar-SA) plus Apple Translation stream straight into a scrollable feed. No network, no waiting.

Layer 2 Background

Async enrichment

Parallel Quran + Hadith specialists feed a deterministic arbiter. When enrichment wins, a plain row upgrades into a citation card — in place, never re-rendering the feed.

04 — Core AI Systems

Four matching systems doing the heavy lifting.

01

Quran specialist

localQuranDB.ts · ~787 lines
  • Full Quran indexed at module load from quran_english.json
  • Arabic normalisation — alef variants, ta marbuta, diacritics, BiDi marks
  • Fuzzy word match with 1-edit tolerance
  • Confusion-class collapse for STT phoneme errors (ق/ك, ص/ث/س, ع/ء…)
  • Composite scoring: overlap, verse coverage, ordered run, cursor bonus, khutbah penalties
  • Precision gates + sub-verse indexing at Quranic stop marks (wuquf)
02

Hadith specialist

hadithSpecialist.ts + localHadithDB.ts
  • Tier 1 — signals: narrator names, reporting verbs, markers, Khutbat al-Hājah openers, buffer continuity
  • Explicit short-circuit triggers for source attribution & narrator+marker
  • Tier 2 — corpus: 15 hand-verified khutbah hadiths (incl. Abu Dawud 2118)
  • Input-coverage-dominant scoring for short quotes from long matns
  • Canonical paint bypasses buffer and Apple/Groq when matched
03

The arbiter

arbiter.ts · deterministic
  • Hadith corpus / explicit triggers beat Quran false positives
  • Clean Quran match beats gray-zone Hadith
  • Higher composite with margin wins; ties default to Quran for recitation symmetry
04

STT layer

speechRecognition.ts
  • Continuous streaming, interim results, fully on-device
  • Idempotent start/restart guards — fixes cascading parallel sessions
  • Coalesced restart with exponential backoff
  • iosVoiceProcessingEnabled: false — the only app-level DSP knob Apple exposes
05 — Key Engineering Decisions

The calls that shaped the build.

  • Parallel specialists + arbiter instead of a sequential if-tree — like three domain experts weighing in on every chunk.
  • Layer 1 never awaits Layer 2 — enrichment mutates chunks in place via updateChunk.
  • On-device first — Quran DB, Hadith corpus, Apple STT and translation; Groq only as a Hadith source backup.
  • STT-aware matching — confusion collapse and ordered runs, not exact string match.
  • Dedup windows — the same verse or hadith is suppressed within a 2-minute window.
  • Real mosque constraints — a Release iOS build runs offline for testing without a laptop or Metro.
