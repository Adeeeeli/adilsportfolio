01 — The Problem

Generic classifieds weren't built for car people.

Qatar's car buyers and sellers were stuck on platforms that treat a Porsche 911 the same as a flat-pack wardrobe — cluttered feeds, shallow filters, and listing flows that don't understand make, model, variant, or spec.

⌇

Static filter lists — engine sizes that don't exist for the car you picked

⊘

Everything-marketplace noise — jobs, property, and distractions mixed in

⚠

Weak listing data — sellers re-type specs the platform should already know

The product constraint

Browse and sell had to feel fast on real devices while carrying a catalog deep enough that a Bugatti Chiron never shows a 1.0L engine option.

02 — The Solution

A car-only marketplace with catalog intelligence.

IMSHI pairs a polished Expo mobile app with a static vehicle catalog served from Firebase Hosting and live listings in Firestore — so filters, pickers, and listing forms always pull from the same source of truth.

04 — Core Product Systems

Four systems that define the experience.

01

Vehicle catalog

catalog/ · 146 make files
  • 49,806 enriched variants — make, model, variant, year, engine, fuel, body
  • Hosted on Firebase Hosting — zero Firestore reads per browse
  • GitHub Actions deploy on push; app re-fetches on version bump only
  • AI-maintainable JSON schema for adding new model years
02

Dynamic filters

vehicleDataService.ts · Homepage · SearchFiltersScreen
  • Engine, fuel, transmission, doors scoped to make/model/variant
  • useMemo + auto-reset when parent selection invalidates a child
  • Listing count index on picker labels — "BMW 3 Series (12)"
  • Prevents impossible combinations before search runs
03

Listing lifecycle

AddVehicleStep1–3 · car-details.tsx
  • Three-step sell flow with catalog-backed pickers and photo upload
  • QAR formatting, share to imshi.app/cars/{id}, map approximate location
  • Contact seller via WhatsApp (pre-filled message) or phone call — no in-app chat
  • Inspection report CTA and recently-viewed persistence
  • 7-day preview + 7-day grace; expired listings hidden in Cloud Functions search
04

Design system

tokens/designTokens.ts · components/ui/
  • Monochrome palette — black chips on #F3F2F1 screens
  • Reusable AppButton, Typography, VehicleCard, NativePicker, drawers
  • Full-screen drawers for make/model — not native dropdowns
  • Custom tab bar, skeleton loaders, haptics on key actions
05 — Key Product Decisions

The calls that shaped IMSHI.

  • Cars only — positioning as an automotive marketplace for enthusiasts, not another horizontal classifieds app.
  • Static catalog on Hosting — 50k+ reference rows never belong in Firestore reads; JSON + CDN beats per-user document charges.
  • Dynamic filters from data — if the catalog has no 1.0L for Bugatti, the UI never shows it.
  • Guest browse first — anonymous auth bridge so marketplace feeds work before sign-up.
  • Drawer-based pickers — full-screen make/model selection matches Figma and scales to 146 makes.
  • OS-native seller contact — WhatsApp and tel: intents instead of building chat infrastructure.
  • Listing expiry as product policy — 7-day preview, 7-day grace, daily cleanup in Qatar timezone.
  • Demo-resilient marketplace — local demo listings + catalog fallback when Firestore rules or auth drift in dev.
